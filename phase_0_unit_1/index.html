FROM PROGRAMMER TO BROGRAMMER
APRIL 20, 2014

If as many people are programming as would seem to be programming, then it would be inexcusable to program alone.

-Tennessee Williams (paraphrased)

Programmers have a bad reputation when it comes to working with others. In many cases this is merited (boy do I have some stories), but the best programmers know how to work in teams. I’m glad that DBC is starting us early on group programming.
I’ve enjoyed working with other DBC students thus far. We’re a pretty fun bunch.
One thing that’s been great about working with others is that we’re able to fill in each others’ gaps in knowledge. When I don’t know something generally another student will, and when someone I’m working with (such as in this week’s GPS) doesn’t know something I’m able to help out.
I noticed that there is definite caution to the way students interact. Everything is posed as a tentative question, such as “Would it be OK with you if I navigated now?” This isn’t a bad thing, it is good that everyone’s polite, but I think knowing that we are constantly under review changes how we interact and adds a bit of pressure.
I’m honestly not crazy about DBC’s feedback system. It is purportedly anonymous, but it is always clear who’s giving you feedback based on the timing. Don’t get me wrong, I’m happy to be critiqued. My discomfort comes more from when I have to write feedback for others. It’s uncomfortable to have to tell others what their character flaws are. In the DBC system we’re required to spend half our review on what the reviewee did wrong. My partner in my first GPS was great, I honestly didn’t want him to change how he interacted with team member. However, I had to come up with something he did wrong per the feedback prompt.
I would prefer it if feedback was left to the instructors. They’re present for the entire session and can focus on the student interactions, freeing us to focus on the assignment.


POPULARITY IS A SYNTACTICALLY-TRANSMITTED-DISEASE
APRIL 20, 2014

Why is JavaScript so popular?

-Dev Bootcamp Prompt

People have been hating on JavaScript for a long time. Anyone who reads Hacker News some semblance of regularity knows that programmers love to write about how much JavaScript sucks. And in many ways they have a point. Compared to other languages like Python and Ruby it is ugly. That’s why languages that compile to JavaScript, like CoffeeScript, exist. They want to make scripts shorter and prettier. I used CoffeeScript for a little while and really liked it, but the problem is that its not the standard. JavaScript is. JavaScript’s biggest strength is that pretty much everyone uses it for almost every website.
Another reason why JavaScript is popular is that its fairly easy to use. Sure, the creators of CoffeeScript would probably disagree, but anyone who has waded through C or VB state machines (the horror!) knows that code can get a lot uglier than JVS.
A big part of usability is tool diversity. There are TONS of tools to program in JavaScript. While this is perhaps a result of popularity (more people using javascript means more customers for people who make programming tools) it also feeds popularity. Programmers, like everybody else, like to have options. The more options available the more incentive there is to learn a language.
Another reason why JavaScript is popular is that it can be edited, modified, and tested really, really quickly. I’ve spent years programming hardware, and testing new code means compiling it in an editor, uploading it to a microprocessor and powering up the device. Depending on the compiler it could take up to a minute to test new code. Programming involves lots of testing, so this ate up a huge amount of time. With JavaScript I can write a new block of code and test it in just a couple seconds.
That’s pretty awesome. JavaScript, for better or worse, will continue to be the standard for the foreseeable future. Many gripe about this, but believe me, it could be a lot worse.


PUTTING THE "CLASS" IN "CLASS"
APRIL 14, 2014

What are the best practices associated with using classes vs. ids?

-Dev Bootcamp Prompt

When I joined my first Guided Pairing Session with Greg and Ariel I relized I had only been using classes up to that point. I had never once used an id. This was technically working for me. None of my css files were too complicated so I could create a class for a single element without any negative consequences. However as Ariel explained to me, this isn't an ideal practice long-term. Ids are for single elements, whereas classes are for groups of them. They are defined slightly differently (ids have hash tags when being defined in CSS) but attributes are specified the same way. I've since found myself using ids a lot more, and I think my code is the better for it.
